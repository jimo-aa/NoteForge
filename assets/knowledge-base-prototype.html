📝 AI 摘要
本文详细描述了 NoteForge 系统的分层架构设计和关键技术选型理由。系统采用微服务架构，后端基于 Spring Boot 3.3 构建，前端使用 React 18 + TypeScript，数据库选用 PostgreSQL 16 并启用 pgvector 扩展以支持向量检索。核心引擎使用 Rust 编写，确保高性能和内存安全。

1. 项目背景

随着业务规模的持续扩大，现有的单体架构已无法满足快速迭代和高可用性的需求。本项目旨在构建一个基于微服务架构的智能知识管理平台，支持多端协同、AI 驱动的知识管理和高效的全文检索。

1.1 业务痛点

当前系统面临以下核心挑战：

  • 架构耦合 — 单体应用中各模块耦合严重，变更影响范围大
  • 搜索效率低 — 仅支持关键词匹配，无法理解语义，准确率不足 60%
  • 知识孤岛 — 不同格式的知识文档分散存储，缺乏统一的管理和检索入口
  • 扩展瓶颈 — 数据库连接和计算资源无法按需扩展

1.2 项目目标

通过架构升级实现以下目标：

  • 系统可用性 ≥ 99.95%
  • 搜索响应时间 P99 < 200ms
  • 支持 1000+ 并发用户
  • 支持多格式文件导入和 AI 自动分类

2. 技术选型

2.1 后端框架

经过对 Spring Boot 3.3、Quarkus 和 Micronaut 的全面评估，最终选择 Spring Boot 3.3 作为后端框架，主要考量：

  • 成熟的生态和社区支持
  • GraalVM 原生编译支持，启动时间 < 100ms
  • 虚拟线程 (Virtual Threads) 大幅提升并发处理能力

2.2 数据库选择

采用 PostgreSQL 16 作为主数据库，并启用以下扩展：

扩展用途版本
pgvector向量相似度搜索0.7.0
pg_trgm模糊文本匹配1.6
uuid-osspUUID 生成1.1

2.3 前端技术

桌面端采用 Tauri 2.x + React 18 技术栈：

2.3.1 React 18

利用 React 18 的并发特性（Concurrent Features）优化渲染性能。使用 Suspense 和 useTransition 实现流畅的交互体验。

2.3.2 TypeScript

全栈 TypeScript 确保类型安全，共享类型定义减少前后端沟通成本。

// 共享类型示例
interface KnowledgeDoc {
  id: string;
  title: string;
  category: string;
  tags: string[];
}

3. 架构设计

3.1 分层架构

系统采用经典的四层架构：

  1. 表现层 — React 前端，负责 UI 渲染和用户交互
  2. API 层 — Spring Boot 网关，负责请求路由和认证
  3. 服务层 — 业务逻辑实现，微服务拆分
  4. 数据层 — PostgreSQL + Redis + MinIO + Tantivy

3.2 微服务拆分

按业务领域拆分为以下微服务：

  • user-service — 用户认证和权限管理
  • note-service — 笔记 CRUD 和同步
  • knowledge-service — 知识文档管理
  • search-service — 全文搜索和语义搜索
  • ai-service — AI 内容提取和 RAG 问答

4. 数据流

用户请求经 API 网关路由到对应微服务，服务间通过 gRPC 通信，异步任务通过 RabbitMQ 消息队列解耦。

5. 部署方案

采用 Docker Compose 编排所有服务，Kubernetes 用于生产环境自动扩缩容。CI/CD 流程基于 GitHub Actions，自动构建和部署。
