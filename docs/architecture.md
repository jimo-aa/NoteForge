## 二、关键设计决策

### 2.1 为什么用 Tauri 而不是 Electron？

| 维度 | Tauri | Electron |
|------|:-----:|:--------:|
| 安装包体积 | **< 5MB** | 150MB+ |
| 内存占用 | **< 50MB** | 200MB+ |
| 启动速度 | **< 1s** | 3s+ |
| Rust 原生引擎 | ✅ 内置 | ❌ 需 Node 桥接 |
| 安全性 | ✅ 隔离沙箱 | ❌ Node 全局访问 |

→ **Tauri 完胜，尤其适合笔记工具这种开机自启、常驻后台的场景**

### 2.2 为什么需要 Rust 核心引擎？

笔记 App 的核心操作（Markdown 解析/搜索/加密）在纯 JS/Python 下性能瓶颈明显：

| 操作 | JS 方案 | Rust 方案 | 提升 |
|------|:-------:|:---------:|:----:|
| Markdown 解析 (10万字) | ~200ms (marked) | ~8ms (pulldown) | **25x** |
| 全文搜索 (1万笔记) | ~500ms (lunr.js) | ~15ms (tantivy) | **33x** |
| AES 加密 (1MB) | ~50ms (WebCrypto) | ~3ms (aes-gcm) | **16x** |
| CRDT 合并 (1000 ops) | ~100ms (yjs) | ~20ms (automerge-rs) | **5x** |

### 2.3 为什么用 PostgreSQL + pgvector 而不是单独部署 Milvus？

- **MVP 阶段**：pgvector 够用，减少运维复杂度
- **规模化后**：Milvus 作为独立向量库，pgvector 兜底
- **数据一致性**：笔记和向量在同一个事务中，保证 ACID

### 2.4 CRDT 同步方案选择

```
❌ Google OT (Operational Transformation)
   - 中心化架构，需要 server 做 transform
   - 难以支持 P2P

✅ CRDT (Conflict-free Replicated Data Types)
   - 去中心化，每个客户端独立 merge
   - 天然支持离线/P2P
   - Automerge-rs 是 Rust 实现的 CRDT 库
   - 与 Yjs 兼容 (Yrs)
```

## 三、安全设计

### 3.1 端到端加密

```
用户设置密码
    │
    ▼
Argon2id 密钥派生 (KDF)
    │
    ├── 主密钥 (Master Key)
    │       │
    │       ├── AES-256-GCM → 笔记内容加密
    │       ├── HMAC-SHA256 → 内容完整性校验
    │       └── 密钥包装 → 服务器存储加密后的密钥
    │
    └── 挑战值 (用于验证密码正确性)

服务器永远不接触明文密钥和明文内容。
元数据（标题/标签/日期）不加密以保证搜索可用。
```

### 3.2 API 安全

```yaml
Rate Limiting:
  - 未认证: 10 req/min
  - 已认证: 100 req/min (一般), 10 req/min (AI)
  - 写入: 30 req/min

Authentication:
  - JWT (access token 15min + refresh token 7d)
  - OAuth 2.0 (Google/GitHub/飞书)
  - API Key (第三方集成)

Encryption:
  - TLS 1.3 (所有通信)
  - 端到端加密 (笔记内容)
  - AES-256 静止加密 (数据库/存储)
```
