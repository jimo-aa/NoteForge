### 1.2 加密架构

```
用户密码
    │
    ▼
Argon2id (memory-hard KDF)
    │
    ▼
AES-256-GCM (authenticated encryption)
    │
    ├── 随机 12-byte nonce (OsRng)
    ├── 加密数据
    └── 16-byte GCM 认证标签
    │
    ▼
hex 编码 → SQLite 存储
```

### 1.3 审计结果

| # | 检查项 | 状态 | 说明 |
|:-:|--------|:----:|------|
| 1 | **对称加密算法** | ✅ 通过 | AES-256-GCM，NIST 推荐标准，提供认证加密 |
| 2 | **密钥派生** | ✅ 通过 | Argon2id (memory-hard, 抗 GPU/ASIC) |
| 3 | **随机数生成** | ✅ 通过 | `OsRng` (系统 CSPRNG) 替代 `thread_rng` |
| 4 | **Nonce 管理** | ✅ 通过 | 每次加密生成唯一随机 nonce，12 字节 |
| 5 | **密钥存储** | ✅ 通过 | 密钥仅存在内存中，从不持久化到磁盘 |
| 6 | **盐值管理** | ✅ 通过 | 随机 16 字节盐值，持久化到 SQLite，防止彩虹表 |
| 7 | **篡改检测** | ✅ 通过 | GCM 认证标签验证，篡改密文会返回错误 |
| 8 | **空数据加密** | ✅ 通过 | 支持空字符串加密/解密 |
| 9 | **大容量数据** | ✅ 通过 | 100KB 内容加密/解密通过 |
| 10 | **无效输入处理** | ✅ 通过 | 非法 hex、短数据、未初始化均返回错误 |
| 11 | **跨会话持久化** | ✅ 通过 | 盐值存储 + 重派生机制已验证 |
| 12 | **不同密钥隔离** | ✅ 通过 | 不同密码生成的密钥无法互相解密 |

### 1.4 发现的问题与修复

| # | 问题 | 严重度 | 修复 |
|:-:|------|:------:|------|
| F-1 | `rand::thread_rng()` 用于密钥/盐生成 | 中 | ✅ 已替换为 `OsRng`（密码学安全 PRNG） |
| F-2 | 最小密文长度检查为 29 字节 | 低 | ✅ 降低为 28 字节（支持空字符串加密） |
| F-3 | 缺少篡改检测测试 | 低 | ✅ 新增 `test_tampered_ciphertext_detected` 测试 |

### 1.5 剩余风险

| 风险 | 等级 | 说明 |
|------|:----:|------|
| 密码强度依赖用户 | 🟡 中 | 应用仅要求 6 字符最小长度，无强度计 |
| 无密钥轮换机制 | 🟢 低 | 加密密钥在会话期间不变（适合个人笔记场景） |
| 盐值存储在明文 SQLite | 🟢 低 | 盐值无需保密（密码学设计），仅需唯一性 |
| 无前向安全性 | 🟢 低 | 密钥泄露可解密历史数据（可通过定期改密缓解） |

## 四、审计结论

**评级：🟢 通过（低风险）**

NoteForge V1.0 的端到端加密实现遵循行业最佳实践：
- 使用 **AES-256-GCM** 认证加密（非对称加密对笔记场景过度设计）
- 使用 **Argon2id** 密码派生（OWASP 推荐的密码哈希算法）
- 密钥严格保持在内存中，不持久化
- 密码学随机数使用 **OsRng**（系统级 CSPRNG）
- 所有依赖漏洞均已修复或门禁拦截

**建议在 V2.0 中改进：**
1. 增加密码强度指示器（UI）
2. 添加密钥轮换/改密支持
3. 考虑集成硬件安全模块（TPM/安全芯片）
